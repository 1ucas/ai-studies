Main Concepts

The vocabulary. If you can't fluently use these words, the rest of the guide reads as noise.

01 Main Concepts — the vocabulary first

Anchor terms you'll see thrown around in the rest of the guide and in the study. Each is 2–4 sentences — enough to use the word fluently without faking it.

1.1 Token

The unit an LLM reads and writes. Roughly a chunk of 3–4 characters in English (a short word, or a piece of a long word). "Hello, world!" is about 4 tokens. Models are priced per token, have context windows measured in tokens, and generate one token at a time. The tokenizer is the (deterministic) algorithm that splits text into tokens — and different tokenizers can change your bill even when the visible text looks identical.

1.2 Transformer

The neural-network architecture that powers every modern LLM (GPT, Claude, Gemini, Llama, DeepSeek, Kimi — all transformers). Introduced in the 2017 paper "Attention Is All You Need." Replaced the older RNN/LSTM approach. The transformer's superpower is that it processes all tokens in a sequence in parallel (instead of one-at-a-time like RNNs), which is what made training on internet-scale data feasible.

A transformer is built out of stacked "blocks." Each block does two things: attention (look at relevant other tokens) and a feed-forward layer (process each token independently). Stack ~30–100 of those blocks, train on trillions of tokens, and you have an LLM.

1.3 Attention

The mechanism inside a transformer that lets each token "look at" other tokens in the sequence and weight how much each one matters. When the model is generating the next word after "The cat sat on the …", attention is what lets it look back at "cat" and "sat" — and weight them more heavily than filler words — when deciding the next token.

Concretely: every token produces three vectors — Query (Q), Key (K), Value (V). The Query of the current token is compared (dot product) against the Keys of all other tokens to produce attention scores. Those scores weight the Values, which get summed → the token's new representation. "Self-attention" means tokens attend to other tokens in the same sequence.

Attention is also the most expensive part of a transformer. Naive attention is O(n²) in the sequence length — double the context, 4× the cost. Variants like Flash Attention, sliding-window attention (Mistral), Multi-Query Attention (PaLM), Grouped-Query Attention (Llama 3), and linear attention (Mamba/RWKV hybrids) all try to reduce that cost. Modern long-context models (Gemini 1M+) almost certainly use a mix of these.

1.4 Context window

The maximum number of tokens the model can see in a single request — input + output combined (or sometimes counted separately, varies by provider). When you hear "Gemini has a 1M context window," it means the model can hold ~1 million tokens of input. Bigger window → more documents/code/conversation history you can shove in without RAG.

1.5 Chain-of-Thought (CoT)

The model's intermediate "thinking" — written out step-by-step before producing a final answer. Introduced as a prompting trick in 2022 ("Let's think step by step…") and now baked into reasoning models as a built-in behavior. Three flavors to know:

  • Prompted CoT — you ask the model to think step by step. Works on any model.
  • Visible reasoning — the model emits step-by-step reasoning into the response (you and the user see it).
  • Hidden / provider-owned "thinking" — used by OpenAI reasoning, Claude extended/adaptive thinking, Gemini Thinking, and DeepSeek V4 Pro thinking mode. The model spends extra generated tokens on internal reasoning; providers may hide it, summarize it, encrypt/sign it, or expose special thinking blocks. You pay for those thinking tokens — that is why reasoning models can cost 5–30× more in output.

Why it works: writing intermediate steps gives the model more "compute" to reach the right answer (it allocates more forward passes to harder questions). Why it costs: you are literally paying for thousands of extra output tokens you never see.

1.6 Mixture of Experts (MoE)

An architectural trick to make models bigger without proportionally more compute per token. Instead of one giant feed-forward block per layer, you have N "expert" blocks plus a small router network that picks 1–2 experts to actually run for each token. The model has a huge total parameter count, but only a small active count per forward pass.

Example: Kimi K2.6 is ~1T total / ~32B active per token — a classic MoE. DeepSeek V4 Pro has a similar ratio (1.6T total / 49B active) but uses DSA (DeepSeek Sparse Attention) rather than traditional MoE. You get the knowledge breadth of a giant model at the inference cost of a much smaller one. Trade-off: all expert weights must reside in VRAM at all times, even those not selected for a given token — so weight residency is higher than a dense model of equivalent compute cost. The flip side is that compute per token is lower (only the active experts run matrix multiplications). Routing can also be uneven: popular experts get overloaded while others sit in VRAM without contributing.

Most cheap open-weight models are MoE in 2026. Closed flagship models (GPT-5.x, Claude 4.x, Gemini 3.x) are widely believed to be MoE but the labs do not confirm architecture details.

1.7 Classifier

A model (often small) whose job is to assign one of a fixed set of labels to an input. "Is this email spam or not-spam?" "Is this user question about billing, technical, or account?" Output is a category (or a probability over categories), not free text.

In LLM systems classifiers do the cheap upstream work: intent detection, routing, safety filtering, relevance scoring. They can be (a) a fine-tuned small model like DistilBERT/Llama-3.1-8B, (b) a frozen LLM prompted to output a single label, or (c) a classical ML model (logistic regression on embeddings). Big LLMs are overkill — a fine-tuned small classifier is faster, cheaper, and usually more accurate on a fixed taxonomy.

1.8 Routing (in LLM systems)

The decision of which model (or which tool, or which downstream component) should handle a given request. "Routing" appears at multiple layers:

  • Model routing — pick which LLM (Haiku vs Sonnet vs Opus). The §03 topic.
  • Tool routing — given a request, pick which tool/function the agent should call.
  • Skill routing — pick which sub-agent or specialized prompt handles this.
  • Expert routing — inside an MoE model, the router network picks which experts fire for a given token (different layer of abstraction — same word).

A "router" is usually a classifier (see 1.7) — that's why the two concepts are joined at the hip.

1.9 Model lab vs Provider vs Inference platform

Three things that often get conflated:

  • Model lab — who trained and owns the model. Anthropic, OpenAI, Google DeepMind, Meta, DeepSeek, Moonshot, Z.ai, Mistral, Alibaba (Qwen).
  • First-party provider — the lab serving its own model via API. api.anthropic.com, api.openai.com, ai.google.dev.
  • Third-party inference platform — a company that hosts other labs' models (usually open-weight) on their infrastructure. Together AI, Fireworks, DeepInfra, Groq, Replicate, OpenRouter.
  • Cloud-vendor reseller — your existing cloud account, with frontier models bolted on. AWS Bedrock, Google Vertex AI, Azure OpenAI Service.

"Different provider models per sub-task" in the routing pattern (§03) means: this step uses an Anthropic-hosted Sonnet; the next uses a Together-hosted Kimi K2.6; the embedding step uses Voyage. Different vendors, one workflow.

Why they matter in practice

  • Procurement. If your company already has a $10M/year AWS commit, Bedrock spend counts against it. No new vendor to onboard — Legal, Finance, Security all say yes faster.
  • Data sovereignty & compliance. Requests/responses stay inside your cloud VPC and region. HIPAA, SOC 2, FedRAMP boundaries are inherited from the cloud account. Going to api.anthropic.com directly often fails a regulated-industry security review; Bedrock does not.
  • Networking. Private VPC endpoints — model calls never traverse the public internet. Big for finance, healthcare, government.
  • Unified IAM / observability. CloudTrail / Cloud Audit Logs records every call. Use the same roles, secrets manager, KMS keys you already use.
  • Multi-model marketplace. One contract, one SDK, dozens of models. Switching models is a config change, not a procurement cycle.
  • Trade-off: new model versions usually land on the lab's first-party API days-to-weeks before Bedrock/Vertex. And pricing on these platforms is sometimes ~5% higher than the direct API.
Key insight: "For enterprise AI, the model is rarely the limiting factor — procurement, data residency, and IAM are. Bedrock and Vertex exist because they let you adopt frontier models without re-doing your compliance posture."

Cost optimization levers (memorize these)

How to read this table

  • MoE → cheap inference, big memory. A 1T/32B MoE (e.g. Kimi K2.6) costs about as much to run per token as a 32B dense model — but you need RAM for all 1T at once. DeepSeek V4 goes further: its DSA (DeepSeek Sparse Attention) achieves similar efficiency without traditional expert routing, making V4 Pro (1.6T/49B) cheaper to serve than its parameter count suggests.
  • Long context = attention variant. When you see "1M context" (Gemini, GPT-5.4/5.5, Opus 4.6/4.7, Sonnet 4.6), the model is not using naive quadratic attention at full cost — it's using cheaper attention/cache strategies. The labs rarely publish exactly what.
  • Reasoning mode is now often a request parameter, not a separate model. Claude 4.x has extended/adaptive thinking, GPT-5.x has reasoning effort, Gemini 3.x has thinking budgets, and DeepSeek V4 exposes thinking/non-thinking modes.
  • Tool-use quality varies more than raw IQ. The benchmark gap between Sonnet 4.6 and Kimi K2.6 on raw reasoning is small. The gap on "follows a 12-step tool sequence without going off the rails" is huge — that is where Anthropic still leads.
  • "GQA" / "MLA" — Grouped-Query Attention (used by Llama, most modern transformers — fewer K/V heads than Q heads → cheaper) and Multi-head Latent Attention (DeepSeek's invention — compresses KV cache, lets them serve 128K context cheaply).

3.4 Routing patterns you should be able to name

  • Cascade / triage routing — cheap model answers; if confidence low or a "hard" classifier flags it, escalate to the expensive model. Common shape: Haiku 4.5 → Sonnet 4.6 → Opus 4.6/4.7. Cuts cost 60–80% with minimal quality loss.
  • Task-typed routing — a classifier (sometimes an LLM, sometimes a fine-tuned small model) tags the request (code, chat, math, vision) and dispatches.
  • Speculative / draft-then-verify — cheap model drafts, expensive model verifies / edits. Used in agentic coding (e.g. Haiku proposes, Sonnet reviews).
  • Mixture-of-Experts at the orchestration layer — same name as in model architectures, but here it means routing to different provider models per sub-task in a workflow.
  • Provider failover — primary on Anthropic, fallback to Bedrock-Anthropic or to OpenAI on outage. Tools: OpenRouter, Portkey, LiteLLM.

3.5 Reasoning vs non-reasoning — when each is worth the cost

Reasoning models (Opus 4.6/4.7 extended thinking, GPT-5.5 Pro, DeepSeek V4 Pro thinking mode) burn many extra output tokens on provider-owned internal thinking. Use them when:

  • The task has a verifiable correct answer — math, code that must pass tests, logical puzzles.
  • The task is multi-step planning where a wrong early step poisons the rest.
  • You can afford 5–30× more output tokens and 2–10× more latency.

Do not use them for chat, summarization, classification, or anywhere the answer is "open-ended" — you pay for thinking you do not need.

3.7 Decision flow (the "out loud" version)

  1. Is the data sensitive / regulated? Yes → open-weight on your infra. Stop.
  2. Does the task fit in < 8K tokens and have a small fixed schema? Yes → Haiku 4.5 / Gemini 3.1 Flash-Lite / DeepSeek V4 Flash.
  3. Does it need to reason multi-step over code or math? Yes → Sonnet 4.6 → Opus 4.6/4.7 / GPT-5.5 Pro / DeepSeek V4 Pro (thinking mode).
  4. Is it a long document (> 200K tokens)? Yes → Gemini 3.1 Pro / Gemini 3 Flash.
  5. Is it multimodal? Yes → Gemini 3.1 Pro first.
  6. Default agentic coding / tool use → Sonnet 4.6.
  7. Then ask: can I cache 80% of this prompt? If yes, the choice gets cheaper by 10×.
Common pitfall: Do not recommend "the smartest model" reflexively. In production, 80% of tokens go to routine work — and using Opus for routing is a costly mistake at scale. The right answer is almost always "I would route — here is the cascade."
Killer line: "There is no best model. There is a portfolio. Frontier for the hard 5%, Sonnet-tier for the agentic 15%, Haiku/Flash for the 80% that is routing, classification and extraction. Cost-aware routing is how you make AI products profitable in 2026."

4.4 Context flow: how input + output feed the next call

Every multi-turn app is either stateful or stateless:

  • Stateful: provider stores prior turns. You pass a conversation/response id such as OpenAI's previous_response_id. Easier, but you depend on provider-side state.
  • Stateless: your app stores the conversation. On each request you send the relevant prior user messages, assistant outputs, tool calls, tool results, and retrieved context.

The safe stateless loop is: user input → model output items → tool results → model final answer → next user input. Do not summarize or mutate provider-owned reasoning/signature blocks in the middle of a tool loop; several APIs require those blocks exactly as returned.

Context rule: The assistant output is not just display text. It may be required protocol state. If you throw away tool-call ids, reasoning items, thought signatures, or content-block ordering, the next API call can fail or reason worse.

4.5 Reasoning / thinking tokens in API responses

Do not assume "thinking" means the same wire behavior everywhere. The common rule is: thinking costs output tokens; the provider decides whether you can see it, summarize it, or only count it.

Usage includes output_tokens_details.reasoning_tokens; some models can return reasoning summaries or encrypted reasoning items. Anthropic Claude thinking: { type: "enabled", budget_tokens: ... } thinking content blocks followed by text blocks; summaries/encryption vary by model/version. With tool use, pass the complete unmodified thinking blocks back for the last assistant message.

4.6 Tool calling is a loop, not magic

The model never executes your function. It emits a structured request; your app validates it, runs the function, then sends the result back so the model can continue.

  1. Declare tools with name, description, and JSON Schema / OpenAPI-style parameters.
  2. Receive tool calls from the model with a tool name, arguments, and call id.
  3. Execute in your app after auth, validation, idempotency checks, and audit logging.
  4. Return tool results using the provider's call id field.
  5. Continue until the model returns final text or another tool call.

4.7 How RAG vectors enter the API

You usually do not send raw vectors to the generation model. Vectors live in the retrieval layer. The model receives the retrieved text, metadata, and citations.

External vector database pattern

  1. Ingest: split documents into chunks; call an embeddings API; store { id, vector, text, metadata } in Pinecone, Weaviate, Qdrant, pgvector, Elasticsearch, etc.
  2. Query: embed the user's question using the same embedding model family.
  3. Retrieve: vector search + BM25/hybrid + rerank top results.
  4. Generate: send the top snippets as normal input text: source title, URL/file id, chunk text, and citation id.
const queryEmbedding = await embeddings.create({
  model: "text-embedding-3-large",
  input: userQuestion,
  encoding_format: "float"
});

const matches = await vectorDb.search({
  vector: queryEmbedding.data[0].embedding,
  topK: 8,
  filter: { tenant_id: tenantId }
});

const context = matches.map((m, i) =>
  `[${i + 1}] ${m.metadata.title}\n${m.metadata.text}`
).join("\n\n");

await client.responses.create({
  model: "gpt-5.4",
  input: `Answer using only these sources:\n\n${context}\n\nQuestion: ${userQuestion}`
});

Hosted file-search pattern

Some providers manage the vector store for you. In OpenAI's Responses API, for example, you upload files into a vector store, then pass a hosted tool config such as { type: "file_search", vector_store_ids: ["vs_123"] }. The model can call the file-search tool, and you can optionally ask the API to include the retrieved results in the response for debugging/evals.

Common pitfall: RAG is not "send embeddings to the LLM." It is "use embeddings to choose text, then send the chosen text to the LLM." The only exception is a provider-managed retrieval tool, where you send a vector-store id and let the provider handle retrieval.

4.8 What to log from every model call

  • Request id from provider headers/response metadata for support and incident debugging.
  • Model version, prompt/template version, tool schema version, and retrieval index version.
  • Usage: input, cached input, visible output, reasoning/thinking tokens, total tokens, latency.
  • Tool trace: tool name, args hash, auth principal, result status, retry count, duration.
  • Retrieval trace: query, embedding model, top-K ids, reranker scores, source ids actually cited.
Production rule: "The model API is an event loop with billing. Treat every response item as typed data, preserve provider-owned reasoning/signature items, and make tools/retrieval observable enough to replay."

06 Memory types in AI agents

Drawn from cognitive science. Four types — know them all.

Short-term / Working / In-context

Everything inside the current context window. Like RAM — gone when the session ends. Cheap, fast, limited.

Examples: current conversation, this turn's tool outputs, scratchpad.

Episodic (long-term)

Specific past experiences with temporal context. "Last Tuesday Lucas asked me to refactor X."

Storage: vector DB for semantic search + event log for ground truth.

Semantic (long-term)

Factual knowledge, decoupled from when you learned it. "Lucas's preferred language is C#."

Storage: structured DB for facts + vector DB for concept embeddings.

Procedural (long-term)

"How to do things." Workflows, tool sequences, learned policies. Often encoded in the system prompt / agent definition itself.

Storage: system prompts, fine-tuned weights, skill files.

Architecture in 2026

  • Vector memory = retrieves semantically similar items.
  • Graph memory = retrieves relationally connected items. Production-grade by early 2026 (was experimental in 2024).
  • Memory compression: the "fundamental cognitive step" of moving short-term → long-term. Filter signal from conversational noise.
  • Frameworks: mem0, Letta (formerly MemGPT), Zep, Redis-based stores.

07 Custom MCP — Model Context Protocol

MCP is the "USB-C for AI" — a standardized protocol (Anthropic, late 2024, now broadly adopted) for letting any LLM client talk to any tool/data source.

If you already know REST APIs, here is the analogy

A REST API exposes resources at URLs (GET /users/42) so that humans writing code can integrate with your system. Each client (web app, mobile app, partner backend) writes its own integration: pick endpoints, parse responses, handle auth, retry on failures.

An MCP server exposes tools, resources and prompts over a standard protocol so that any LLM client can integrate with your system without writing client code. The LLM discovers what is available, reads the schemas, and decides at runtime which tool to call.

Same idea — "expose capabilities so something else can call them" — but the caller is an LLM agent, the discovery is automatic, and the contract is uniform across every AI tool in your stack (Claude Desktop, Claude Code, Cursor, Windsurf, Codex, etc).

Mental model: REST is "an API for code." MCP is "an API for LLMs." You can absolutely wrap an existing REST API in an MCP server — and that is exactly what most production MCPs do (the Jira MCP, GitHub MCP, Linear MCP are all thin wrappers over their respective REST/GraphQL APIs, repackaged with LLM-friendly descriptions and schemas).

What MCP adds that REST does not

  • Self-describing tools. The LLM asks "what can you do?" and gets back tool names, descriptions, and argument schemas it can validate against. No human reads OpenAPI first.
  • One integration → every AI client. Build the Jira MCP once; Claude Desktop, Claude Code, Cursor, Codex, Windsurf all use it.
  • Resources are LLM-aware. A resource is a "thing the LLM can pull into context" — not just "data returned from a call." The protocol distinguishes that.
  • Prompts as a first-class primitive. The server can ship reusable prompt templates (e.g. /review-pr) — REST has no analog.
  • Session and notifications. Server can push updates (e.g. "this resource changed, refresh"). REST needs WebSockets or polling for that.

Three primitives an MCP server can expose

  • Tools — functions the LLM can call (write actions, queries).
  • Resources — file-like read-only data the client can fetch.
  • Prompts — pre-written templates users can invoke.

Transport

  • stdio — local subprocess. Used for desktop apps (Claude Desktop, Cursor, Claude Code locally).
  • Streamable HTTP — remote, scalable. For cloud-deployed servers, multi-tenant. It replaced the older standalone HTTP+SSE transport; SSE may still appear inside Streamable HTTP for server-to-client streaming.
  • Wire format: JSON-RPC 2.0 over either transport.

08 RAG — Retrieval Augmented Generation

"Pre-pend relevant context to the prompt." The naive version fails 40% of the time at retrieval — the 2026 stack is hybrid + reranked + sometimes graph.

The modern stack

  1. Chunking — semantic / sliding-window chunks (300–800 tokens). Track parent doc.
  2. Embedding — dense vectors (OpenAI text-embedding-3-large, Cohere, BGE, Voyage).
  3. Hybrid retrieval — BM25 keyword + vector similarity in parallel, fused with Reciprocal Rank Fusion (RRF). The single biggest quality win over naive.
  4. Reranking — top-50 → cross-encoder reranker (Cohere Rerank, BGE-reranker) → top-5. Highest-ROI step. +25–40% precision.
  5. Generation — pass top-K + query to LLM. Cite sources.
  6. Eval — RAGAS metrics (faithfulness, context precision, answer relevance).

When RAG vs alternatives

  • RAG — knowledge changes often, large corpus, need citations.
  • Long context (Gemini 1M, etc.) — small enough corpus to fit; cost permits.
  • Fine-tuning — style/format, not facts. Don't use fine-tuning for fresh data.
  • GraphRAG — multi-hop questions over a structured domain.
Bumper sticker: "In 2026, retrieval is the bottleneck, not generation. Hybrid + rerank is the default." Don't reach for fancier (HyDE, multi-query, agentic RAG) until you've measured RAGAS scores and proved you need it.

09 Grafo de Pesquisa (Research Graph)

"Research graph" shows up in two related contexts — make sure you can distinguish them.

1. Deep Research agents (the "research graph" as workflow)

An agent that iteratively plans → searches → reads → synthesizes, building a graph of sub-questions where each node is a search query/document and edges encode "this finding led to that follow-up." Google's Deep Research, OpenAI Deep Research, and Anthropic's Research feature all follow this pattern.

  • Long-running (minutes, not seconds).
  • Uses test-time compute — many model calls to refine.
  • Connects to web + MCPs + uploaded files.
  • Output is typically a long structured report with citations.

2. GraphRAG (the "research graph" as data structure)

Instead of retrieving flat chunks, you build a knowledge graph from your corpus (entities + relationships) and retrieve connected subgraphs at query time. Better for multi-hop questions ("which authors collaborated with X on topic Y between 2020-2024?") that flat RAG fails on.

  • Microsoft GraphRAG popularized this in 2024 — by 2026 production-grade.
  • Deep GraphRAG, FastInsight are current research frontier — hierarchical retrieval, semantic-topological reranking.
  • Vector RAG retrieves semantically similar facts; graph RAG retrieves relationally connected facts.
If asked "what's a research graph": Lead with the agent workflow (search graph as a thinking trace), then mention GraphRAG as the data-side counterpart. Bonus: mention that in both cases the graph is a DAG at execution time (no cycles, dependencies flow forward).

10 DAG — Directed Acyclic Graph

A graph of nodes + directed edges with no cycles. In AI agent land, nodes are LLM calls / tools / retrievers / branches; edges are dependencies. No cycles = execution always terminates.

Why DAGs for agents

  • Explicit data flow — easy to reason about, log, replay.
  • Parallel execution where edges allow.
  • Deterministic scheduling, retries per node, partial failure recovery.
  • Easy observability — every node is a checkpoint.

Frameworks

  • LangGraph — DAG-based agent orchestration in the LangChain ecosystem. State + nodes + edges. The reference.
  • LlamaIndex Workflows — similar pattern, event-driven.
  • Haystack (deepset) — pipeline DAGs.
  • Inngest / Temporal / Prefect / Airflow — general-purpose DAG schedulers being adopted for AI.

2026 nuance

Pure DAGs are restrictive — real agents often want conditional cycles (re-plan, retry). LangGraph supports cycles but encourages bounding them with iteration limits. The pattern is "DAG with controlled loops" rather than free-form recurrence.

Adaptive multi-agent systems can now restructure their DAGs at runtime based on intermediate results — this is a hot research area.

How to build an eval suite (production pattern)

  1. Define the task + success criteria. Not "be helpful" — "extract the order ID from emails with ≥98% accuracy."
  2. Seed dataset from real production traces. Hand-label 50–200 examples covering golden path + edge cases.
  3. Pick a grader:
    • Exact match / regex (cheapest, brittlest)
    • Code-based (assertions, JSON schema)
    • LLM-as-judge (flexible, biased — pair with rubric + calibration)
    • Human review (gold but slow)
  4. Run in CI on every prompt/model change. Block merges on regressions.
  5. Auto-grow from failures: every production incident becomes a new test case — the regression suite is your moat.
  6. Layer it: automated evals pre-launch → A/B test in prod → user feedback loop → distribution drift monitoring.

How to build a benchmark (rarely what you want)

  1. Define a capability, not a product — "reasoning over chains of arithmetic" not "our checkout bot."
  2. Collect public, hold-out dataset. Document contamination policy (was the model trained on it?).
  3. Single objective metric. Reproducible scripts. Versioned.
  4. Publish — it only matters if others run it.

Key tools (2026)

  • OpenAI Evals, DeepEval, Promptfoo, Braintrust, LangSmith, Inspect (UK AISI).
  • RAGAS for RAG-specific metrics (faithfulness, context precision, answer relevance).
The killer line: "Benchmarks tell you which model to start with. Evals tell you when you're shipping a regression." If you only remember one thing about this section, remember that.

13 Spec-Driven Development (SDD)

"Inversion of the power structure: specifications no longer serve code — code serves specifications." Specs become executable artifacts that gate AI codegen.

The four (sometimes five) phases

  1. Constitution — persistent project principles: tech stack, coding standards, security rules. Immutable context for every AI run.
  2. Specification — unambiguous user stories, acceptance criteria, constraints. Intent only — no implementation.
  3. (Clarification) — structured pass to surface unspecified edge cases. Q&A logged back into the spec.
  4. Architecture / Plan — human-in-the-loop technical plan. Tech choices, module boundaries, data model. You review and sign off.
  5. Tasks — break the plan into discrete, AI-executable tasks. Modeled as a DAG (Task Graph): dependencies explicit, parallelism where possible.
  6. Implementation — AI agent walks the task DAG, generates code, runs tests, marks tasks complete.

Why the Task Graph is a DAG

  • Tasks have dependencies (B needs A's schema, C and D can run in parallel).
  • No cycles means the agent can schedule work — execute leaves first, work up.
  • Failure isolates to a subtree. You can re-run one task without losing the others.
  • This is exactly how build systems (Bazel, Nx) have always worked — SDD borrows the pattern.

Tools (2026)

  • GitHub Spec Kit — open-source toolkit, the reference implementation. Has /specify, /plan, /tasks commands.
  • Kiro (AWS) — GA Nov 2025. VS Code fork with deep SDD integration.
  • Tessl — spec-as-the-source-of-truth approach.
  • Claude Code, Cursor — increasingly bake in spec-driven workflows.
The essential framing: "Traditional specs are documents humans read. SDD specs are contracts the AI executes against — they gate codegen, drive task graphs, and survive across model versions. The spec is the durable artifact; the code is a refreshable build product."

Where it intersects everything else in this guide

  • The Task Graph is a DAG (§10).
  • Each task often runs through a research graph (§09) — read related code, find prior patterns.
  • Evals (§11) become the executable acceptance criteria.
  • Prompts driving each phase are versioned (§12).
  • The agent uses MCP (§07) to talk to git, the build, the test runner.
  • Context7 (§14) feeds it the actual library APIs to use.

15 What is Context7?

Context7 is an MCP server (built by Upstash) that serves up-to-date, version-specific library documentation into your AI coding tools — Cursor, Claude Code, Windsurf, OpenCode, anything that speaks MCP.

The problem it solves

LLMs hallucinate APIs because their training cutoff lags real library releases. Context7 fetches fresh docs at query time so the model sees the actual current API rather than reciting a 2-year-old version from memory.

How it works

  • Two tools exposed via MCP:
    • resolve-library-id — map "react query" to a Context7 ID.
    • query-docs / get-library-docs — fetch version-specific docs by ID + query. The visible name depends on the client/package wrapper.
  • Free tier, MIT-licensed server, optional API key for higher rate limits.
  • Hosted at context7.com.

Why Upstash?

Upstash is a serverless-Redis / messaging company. They built Context7 as a developer-tool play and run it on their own infra — that's why they can offer a free tier sustainably.

Use case: Any time you ask Claude/Cursor to use a library and it generates calls to a function that doesn't exist or moved — that's Context7's pitch. Add "use context7" in your prompt and the tool will pull current docs.
